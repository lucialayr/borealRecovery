## Coding Style Guidelines

### R Code Standards
- Use `=` for assignment (NOT `<-`)
- Use `snake_case` for names
- Minimal line breaks (except for %>%, +, {}, theme())
- Prefer tidyverse functions
- Minimal comments (code should be self-documenting)
- Lines under 80 characters

### Repository Design Philosophy
- Clear separation: data / scripts / figures
- Pattern: `*_final.R` produces final data → `*_plot.R` produces plots
- Balance: avoid redundancy vs. keep data manageable vs. minimize RAM usage
- Each figure has its own pipeline where possible
- Avoid rerunning computationally intensive steps

### Subset Strategy Options (To Discuss with User)

**Option A: Spatial Sampling (RECOMMENDED)**
- Select N representative grid cells (stratified by ecoregion/latitude)
- Keep all patches, years, PFTs for those cells  
- Reduces data by factor of (total_cells/N)
- Maintains temporal patterns and patch dynamics within selected cells
- **Pros:** Preserves local dynamics, easier to validate
- **Cons:** May miss spatial patterns across full region

**Option B: Patch Sampling**
- Sample X% of patches from each grid cell
- Keep full time series for sampled patches
- **Pros:** Maintains spatial coverage
- **Cons:** Loses patch co-occurrence patterns, harder to validate

**Option C: Hybrid**
- Select subset of grid cells (spatial)
- Within each, sample patches if still too large (statistical)
- **Pros:** Best balance of size reduction and representativeness
- **Cons:** Most complex to implement

**Initial Recommendation:** 
Start with Option A using 15-20 grid cells stratified by:
- Ecoregion (to maintain ecological diversity)
- Latitude (to maintain climate gradient)
- Disturbance history (ensure range of disturbance frequencies)

This should reduce database from 25GB to ~1-2GB while maintaining scientific validity.

### Session 1 Continued - Picontrol Naming Bug & Complete Subset

#### CRITICAL BUG DISCOVERED: Picontrol Table Naming Inconsistency
**Problem:** Database has inconsistent table naming between scenarios
- Picontrol tables: `picontrol_d150_npp_[metric]` (has extra "npp" infix)
- SSP tables: `ssp126_d150_[metric]`, `ssp585_d150_[metric]`

**Root Cause:** Historical database structure difference - picontrol includes NPP in naming

**Solution Implemented:** Created `get_table_name()` helper function
```r
get_table_name = function(scenario, variable) {
  if (scenario == "picontrol") {
    return(paste0(scenario, "_d150_npp_", variable))
  } else {
    return(paste0(scenario, "_d150_", variable))
  }
}
```
